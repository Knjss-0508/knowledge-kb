# 第三部分知识转写与候选知识对接说明

更新时间：2026-07-15

## 1. 一句话说明

我负责的是：

```text
方向二结构化数据
→ 数据清洗与证据分流
→ 主题聚类
→ 质检标准检索
→ MiMo/规则转写
→ 模型初标
→ 生成供人工复核的主题候选知识
→ 导出人工反馈和训练样本
```

我不负责 cz 正式知识库中的最终审核和发布。最终发布仍由 cz 现有知识库流程负责。

## 2. 当前负责边界

### 我负责

- 接收方向二导出的 Excel 或结构化记录。
- 过滤产品类型，目前第一期只处理 `产品类型=手机`。
- 清洗核心问题、判定结论、判定依据、分类和图片链接。
- 判断记录是否具备可复用的证据。
- 将同主题的多条记录聚合为主题。
- 从 cz 导出的手机质检标准主表中检索相关标准。
- 调用 MiMo 生成主题级知识草稿。
- MiMo 不可用或失败时，使用规则型兜底草稿。
- 对转写草稿做模型初标和硬规则校验。
- 生成 `topic_review_queue.xlsx` 供人工复核。
- 导出 13 列候选知识、人工反馈和训练样本。

### cz 负责

- 维护正式质检标准主表。
- 提供标准分类、标准版本和标准引用。
- 人工复核候选知识。
- 选择 cz 正式分类和知识层级。
- 进行最终知识库审核。
- 发布正式知识。
- 维护正式知识的生命周期。


其中以下字段会影响候选生成：

```text
核心问题

判定依据
产品类型
一级分类
二级分类
聊天内容
图片链接
```

### 4.2 cz 质检标准主表

标准目录可以是 Excel 或 JSON，主要字段包括：

```text
标准ID / standard_id
主标题 / title
一级分类 / category_l1
二级分类 / category_l2
检索关键词 / keywords
适用范围 / scope
参考话术 / response_snippet
状态 / status
版本 / version
```

只有状态有效的标准会参与检索。转写结果中的标准引用只能来自本次检索结果，不能由模型自由编造。

## 5. 处理逻辑

### 5.1 数据清洗和过滤

1. 读取方向二 Excel。
2. 按产品类型过滤，目前默认只保留 `手机`。
3. 清理空值、重复图片链接和格式不一致的字段。
4. 如果缺少核心问题、判定结论、判定依据或分类，记录会被排除。
5. 排除原因会写入审计结果和工作簿。

### 5.2 证据分流

记录会按证据情况分流：

| 情况 | 处理 |
|---|---|
| 有完整聊天内容 | 可以进入主题候选 |
| 没有聊天但有可用图片 | 可以进入主题候选 |
| 没有聊天且没有可用图片 | 只进入证据缺口，不生成独立知识 |
| 主题只有 1 条有效证据 | 进入待聚合队列，不直接进入人工审核 |
| 主题至少 2 条有效证据 | 可以进入主题级候选 |

图片处理限制：

- 每条记录最多处理 4 张图片。
- 支持公共 JPEG、PNG、WebP 图片。
- 单张图片最大 5 MB。
- 图片下载失败时不能单独形成确定性结论。

### 5.3 主题聚类

不是“一条工单生成一条知识”，而是先根据以下特征聚合相似记录：

```text
问题意图
对象/部位
异常现象
解题方式
一级分类
二级分类
标准关键词
```

主题级候选的目的，是从多条相似案例中沉淀一条可复用知识，避免把单个工单的个案结论直接外推成通用规则。

### 5.4 标准检索

每个主题会先检索相关手机质检标准，最多保留 Top 5 条用于转写和独立初标。

检索依据包括：

```text
问题意图
对象/部位
异常现象
解题方式
一级/二级分类
标准关键词
```

转写阶段和模型初标阶段会分别进行标准检索，用于检查标准引用是否一致。

### 5.5 主题转写

主题转写使用 MiMo 或规则兜底。

MiMo 主题转写输出的是一个 JSON 对象，主要字段为：

```json
{
  "title": "主题级知识标题",
  "subtitles": ["可检索的自然问法"],
  "content": "可复用知识正文",
  "category_l1": "一级分类",
  "category_l2": "二级分类",
  "layer": "L2",
  "knowledge_form": "流程方法",
  "standard_refs": ["标准引用"],
  "applicable_scope": "适用范围",
  "confidence": 0.82,
  "reasoning_summary": "给审核人的简短依据",
  "needs_human_review": true,
  "image_evidence_summary": "图片证据说明",
  "requires_images": false,
  "image_usage_instruction": ""
}
```

`knowledge_form` 只有两个值：

```text
流程方法
具体判定
```

规则约束：

- 没有明确边界、可信标准和充分证据时，只能输出 `流程方法`。
- 外观、显示、拆修、胶状物、功能异常等图片相关问题，默认输出核验流程。
- 机型/型号问题输出查询和核对流程，不输出某个具体机型结论。
- 没有可信标准时，`standard_refs` 必须为空，并标记重点人工复核。
- 标题不能是关键词堆砌，不能使用斜杠串联多个问题。
- 正文不能直接复制原始工单叙述。

### 5.6 模型初标

模型初标和主题转写是两个独立阶段。

模型初标只审核转写结果，不改写转写正文。主要检查：

```text
转写是否合格
标准引用是否一致
证据是否充分
正文是否覆盖规则、步骤和限制条件
图片是否确实必要
标题是否清楚
是否需要重点复核
```

可能的初标结论：

```text
通过
需修改
驳回
证据不足待补充
```

### 5.7 人工复核

人工复核人员可以编辑最终 13 列候选知识，并填写：

```text
审核结论
错误类型
错误原因
审核备注
是否进入训练集
审核人
审核时间
cz 分类
知识层级
```

只有以下两种结论允许进入下一步：

```text
通过
修改后通过
```

## 6. 候选知识格式

### 6.1 cz 主题审核工作簿

cz 主题审核模块应接收：

```text
topic_review_queue.xlsx
```

主工作表必须叫：

```text
topic_review_queue
```

主工作表包含：

```text
主题元数据
模型转写信息
模型初标信息
最终 13 列候选知识
人工复核字段
```

至少需要这些字段才能导入一条主题候选：

```text
主题ID
主标题
知识内容
```

该工作簿还会包含以下辅助 Sheet：

```text
topic_source_mapping       主题与原始记录的追溯关系
topic_model_drafts          原始主题转写草稿
evidence_gap_rows           证据不足记录
pending_cluster_rows        样本不足、暂不审核的主题
excluded_rows               被排除的记录
guide                       工作簿说明
```

### 6.2 13 列知识主表

13 列候选知识字段为：

```text
主标题
副标题
知识内容
知识分类
知识来源
关联标准项
适用范围
生效状态
来源版本
变更类型
失效原因
检索关键词
校验备注
```

当前映射：

```text
流程方法 → 知识分类=检测方法
具体判定 → 知识分类=场景判定
知识来源   = 方向二主题候选
生效状态   = 待审核
变更类型   = 新增
```

## 7. 文件用途

| 文件 | 用途 | 是否上传 cz 主题审核 |
|---|---|---|
| `review_queue.xlsx` | 旧版单条记录审计和模型追踪 | 否 |
| `topic_review_queue.xlsx` | 主题级审核工作簿 | 是 |
| `candidate_knowledge.xlsx` | 未审核的 13 列候选知识 | 否 |
| `candidate_knowledge_for_submission.xlsx` | 本地复核通过后的 13 列结果 | 不是主题审核导入文件 |
| `topic_feedback.jsonl` | 人工反馈事件 | 否 |
| `topic_training_samples.jsonl` | 训练样本 | 否 |
| `data/phone_mvp.db` | 本地审计数据库 | 否 |

特别注意：

```text
上传 cz“主题审核”时，只上传 topic_review_queue.xlsx。
```

## 8. 当前运行方式

### 命令行

安装：

```powershell
pip install -e .
```

生成主题审核工作簿：

```powershell
answer-hub ingest `
  --source "方向二数据.xlsx" `
  --standards "手机-RAG知识库主表.xlsx" `
  --product-type "手机" `
  --output-dir ".\outputs\phone"
```

不调用 MiMo、只跑规则兜底：

```powershell
answer-hub ingest `
  --source "方向二数据.xlsx" `
  --standards "手机-RAG知识库主表.xlsx" `
  --product-type "手机" `
  --rule-only `
  --output-dir ".\outputs\phone"
```

启动本地 Streamlit 工作台：

```powershell
python -m pip install streamlit openpyxl
python -m streamlit run streamlit_app.py
```

### MiMo 配置

只通过环境变量或 `.env` 配置，不写入代码、Excel 或浏览器：

```text
MIMO_API_KEY
MIMO_BASE_URL
MIMO_MODEL
MIMO_TIMEOUT_SECONDS
ANSWER_HUB_DB_PATH
```

如果 MiMo 未配置或调用失败，系统仍然可以使用规则型候选，但会强制标记人工重点复核。

## 9. cz 对接方式

当前阶段统一采用“全量审核”路线：所有方向二数据先生成主题候选，进入 cz 主题审核模块，由人工完成复核后再提交 cz 终审。

后续系统稳定、字段和审核规则确认后，再评估切换到已转写候选直传 API。两条路线当前不并行使用。

### 当前路线：主题审核工作流

```text
方向二原始数据
→ cz/第三部分工作流生成主题候选
→ 导入 topic_review_queue.xlsx
→ 人工复核
→ 选择 cz 分类和 L2/L1/L3
→ 提交 cz 终审
→ 在正式知识库中创建 review 状态知识
→ cz 原有审核流程发布
```

相关接口：

```text
POST /api/v1/topic-candidates/import
POST /api/v1/topic-candidates/{id}/review
POST /api/v1/topic-candidates/{id}/submit-to-cz-review
```

主题审核导入接口当前要求：

- 文件必须是 `.xlsx`。
- 工作表必须存在 `topic_review_queue`。
- 每条记录至少需要 `主题ID`、`主标题`、`知识内容`。
- 已人工复核或已提交终审的主题不会被重复导入覆盖。

当前仓库中的 cz 后端路由已经写入，但 Docker、数据库迁移、真实工作簿导入和浏览器验收尚未在本机完成验证。

### 后续路线：已转写候选直传 API

```text
上游系统完成聚合、标注和转写
→ POST /api/v1/integration/knowledge-candidates:batch
→ cz 查重和向量处理
→ 创建 review 状态知识
```

该接口中的 `knowledge.content` 是已经转写完成的知识正文，不是让 cz 再转写的输入。

它适合后续“上游已经完成转写，只需要把候选知识送入知识库”的场景。

当前阶段暂不使用该路线，也不要求 cz 同时支持两种入口。

### 11.1 当前接入路线

### 11.2 上传文件格式

- cz 主题审核是否只接受 `topic_review_queue.xlsx`？
- 是否强制要求 Sheet 名为 `topic_review_queue`？
- 是否要求所有辅助 Sheet 都存在，还是只读取主 Sheet？
- 哪些列是必填，哪些列可以为空？
- `主题ID` 是否作为重复导入和更新的唯一键？

### 11.3 字段映射

- `知识分类` 是否需要映射成 cz 的 `category_id`？

## 12. 对 cz 的核心说明

可以直接这样说明：

> 当前阶段第三部分负责把方向二数据沉淀成主题级候选知识，所有候选都先进入 cz 主题审核，由人工全量复核后再提交 cz 终审。系统不会把单条工单直接当成正式知识，而是先做清洗、证据分流、主题聚类和标准检索，再由 MiMo 或规则生成转写草稿，最后经过模型初标和人工复核。等字段、标准、分类和审核规则稳定后，再评估使用已转写候选直传 API。
