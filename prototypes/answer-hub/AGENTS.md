## 用户与沟通

- 每次回复开头称呼用户“唔哉”。
- 默认使用简体中文。
- 用户是编程小白，使用简单语言解释，不只给出技术术语。
- 执行代码修改前，先说明：
  1. 准备修改哪些文件；
  2. 问题或需求的判断；
  3. 3～6条操作计划；
  4. 准备运行的命令及原因。
- 不确定文件、接口、配置或业务规则时，先搜索仓库，禁止凭空猜测。
- 除非确实无法安全判断，否则优先采用合理假设继续工作，不频繁打断用户。

## 项目说明

这是“答疑中台知识库”项目，主要包含：

- `streamlit_app.py`：Streamlit 工作台入口；
- `src/answer_hub/`：核心业务逻辑；
- `tests/`：pytest 自动化测试；
- `scripts/`：启动、验收、打包和运维脚本；
- `cz-knowledge-kb/knowledge-kb-master/`：本地 CZ 子项目；
- `data/`、`outputs/`：数据库、真实数据和运行结果。

开始重要任务前，按以下顺序阅读当前说明：

1. `START_HERE.md`
2. `AI_HANDOFF_LATEST.md`
3. 与任务相关的 Runbook
4. `README.md`

旧的 `PROJECT_HANDOFF.md` 和历史 `handoff/` 文件不得作为当前状态的主要依据。

Windows 下读取和写入中文文件时使用 UTF-8，避免产生乱码。

## Skills 使用规则

根据任务自动选择合适的 Skills，并在开始前简短说明使用了哪些 Skills。

- Streamlit 页面、布局、表格、状态或性能修改：
  使用 `developing-with-streamlit`。
- Bug、报错、卡住或性能回退：
  使用 `diagnosing-bugs`。
- 新功能或行为变化：
  优先使用 `tdd`，先写或更新测试。
- 需求不清楚或业务边界不明确：
  使用 `grill-with-docs`。
- 外部 API、框架或模型文档研究：
  使用 `research`，优先查官方资料。
- 修改完成后：
  使用 `code-review` 检查实现和需求是否一致。

不要为了使用 Skills 而使用 Skills。小任务只选择必要的流程。

## 修改原则

- 优先小范围、容易审查的修改。
- 禁止未经要求进行大规模重构。
- 修复 Bug 时先复现，再提出假设、实验和最小修复。
- 保持现有代码风格、目录结构和接口兼容性。
- 可复用的业务逻辑优先放入 `src/answer_hub/`，不要继续把大量业务逻辑堆入 `streamlit_app.py`。
- 不修改与当前任务无关的代码。
- 不随意新增生产依赖；确需新增时先解释必要性和影响。
- 使用明确的错误处理和类型标注。
- 注释只解释不明显的设计意图。
- 未经明确要求，不执行提交、推送、强制重置、清理分支等 Git 操作。
- Git 操作前先确认真正的仓库根目录。

## 业务红线

修改工作流时必须保持以下行为：

- 系统流程只到 CZ 待审核，不自动发布知识。
- 上游的核心问题、判定结论、判定依据和分类字段只作为弱参考与审计信息。
- 当前批量流程不得主动读取、检索或生成标准关联。
- 新候选的“关联标准项”默认保持为空。
- 已有标准关联值必须保留，不得覆盖。
- 低置信度、模型降级或风险候选必须进入人工审核。
- MiMo 不可用时允许规则降级，但必须明确标记为人工优先审核。
- 不得绕过 Qwen3 重复检测和 CZ 终审。
- 批量处理中单条失败不得破坏后续成功项目。
- 不得把测试数据或调试结果误写入正式数据。

## 数据与密钥安全

- 禁止读取、输出、复制或提交 `.env` 中的实际值。
- 只允许参考 `.env.example` 中的变量名称。
- 禁止在代码、日志、测试或文档中写入 API Key、Token、密码或私钥。
- `data/` 中的数据库和 Excel 默认视为真实或敏感数据。
- `outputs/` 中的运行结果默认不得手工修改。
- 测试优先使用 `tests/` 中的 fixture、临时目录和脱敏样本。
- 未经用户明确允许，不调用真实 MiMo、CZ、Qwen3 或其他外部服务。
- 自动化测试默认使用 mock、fake 或本地隔离环境。
- 不新增遥测、分析上报或不必要的网络请求。
- 不修改 `.venv/`、`node_modules/`、`__pycache__/`、`.pytest_cache/` 等生成目录。

## 测试规则

修改前先运行最快、最相关的测试，修改后再次运行。

单个相关测试：

```powershell
.\.venv\Scripts\python.exe -m pytest -q tests\test_相关模块.py
```
